8.1.2 Jezik TypeScript

Angular aplikacijo bomo razvijali v jeziku TypeScript, ki je nadrejena množica jezika JavaScript in ga je možno vanj tudi prevesti. Ključna prednost jezika TypeScript pred jezikom JavaScript je vpeljava podatkovnih tipov, s pomočjo katerih si močno olajšamo razvoj spletnih aplikacij. Večino posebnosti jezika TypeScript bomo spoznali sproti pri razvoju demo spletne aplikacije, na tem mestu zgolj povzemamo različne podatkovne tipe, ki so na voljo v jeziku TypeScript:

  • string predstavlja besedilni niz,
  • number je poljubna številčna vrednost, kjer cela in decimalna števila obravnavamo na enak način,
  • boolean je logična vrednost (true ali false),
  • object predstavlja vrednost, ki ni enostaven podatkovni tip,
  • array predstavlja polje vrednosti danega podatkovnega tipa,
  • enum nam omogoča dodajanje prijaznih oznak množici numeričnih vrednosti,
  • any je poljuben podatkovni tip, tako kot je privzeto pri jeziku JavaScript,
  • void predstavlja pomanjkanje podatkovnega tipa in se običajno uporablja pri funkcijah, ki ne vračajo ničesar.

8.1.4.1 Privzeti modul

Glede na to, da je aplikacija sestavljena iz modulov, je smiselno pogledati, kako je opredeljen privzeti modul AppModule v datoteki app.module.ts.

angular/src/app/app.module.ts

import { NgModule } from "@angular/core";
import { BrowserModule } from "@angular/platform-browser";

import { AppComponent } from "./app.component";

@NgModule({
  declarations: [AppComponent],
  imports: [BrowserModule],
  providers: [],
  bootstrap: [AppComponent],
})
export class AppModule {}

Pri opredelitvi modula opazimo, da najprej poskrbimo za uvoz obstoječih Angular modulov (npr. NgModule in BrowserModule), ki jih bo naša aplikacija uporabljala. Podobno uvozimo tudi komponento AppComponent, ki je vhodna točka naše aplikacije in je podrobneje opredeljena v datotekah s predpono app.component.

Pri opredelitvi elementov Angular aplikacije uporabljamo dekoratorje (angl. decorator), ki predstavljajo načrtovalske vzorce za določanje delovanja prej omenjenih elementov. Angular podpira več vrst dekoratorjev, in sicer za opredelitev razredov, lastnosti, metod in parametrov.

8.1.4.2 Privzeta komponenta

Privzeta komponenta je opredeljena v datotekah s predpono app.component, in sicer:

  • app.component.css opredeljuje stil komponente,
  • app.component.html predstavlja HTML predlogo oz. opredeljuje izgled komponente in
  • app.component.ts opredeljuje logiko delovanja komponente v jeziku TypeScript.

Opomba: Ker v naši aplikaciji uporabljamo poenostavljeno opredelitev komponent, sta stil in HTML predloga kar neposredno vključena v opredelitev komponente v datoteki app.component.ts in datoteki app.component.css ter app.component.html nista prisotni.

V nadaljevanju je prikazan primer HTML predloge v okviru atributa template v datoteki app.component.ts, kjer lahko opazimo, da je način združevanja podatkov iz spremenljivk in HTML predloge zelo podoben generiranju HTML vsebine s pomočjo Handlebars na strani strežnika.

angular/src/app/app.component.ts

@Component({
  selector: "app-root",
  template: `
    <!--The content below is only a placeholder and can be replaced.-->
    <div style="text-align:center" class="content">
      <h1>Welcome to {{ title }}!</h1>
      <span style="display: block">{{ title }} app is running!</span>
      ...
    </div>
    ...
  `,
  styles: [],
})

Poslovna logika komponente se nahaja v datoteki app.component.ts. Najprej se iz Angularja uvozijo podatki, potrebni za delovanje komponente. Potem se komponenti posredujejo informacije, ki jih potrebuje za delovanje (tj. selector, template in styles) in nato se komponento izvozi v obliki razreda.

angular/src/app/app.component.ts

  selector: "app-root",

    ...
  `,

Pomemben podatek je selector, ki je nastavljen na app-root, kar je tudi edina HTML oznaka v začetni index.html datoteki. Prav tako lahko opazimo, da komponento AppComponent izvozimo in to komponento potem tudi poženemo v privzetem modulu app.module.ts (tj. bootstrap: [AppComponent]).

8.1.5 Generiranje Angular aplikacije v jeziku JavaScript

Trenutni spletni brskalniki ne podpirajo jezika TypeScript, ampak zgolj jezik JavaScript. Uporaba jezika TypeScript pri razvoju Angular aplikacij zagotovo pomaga pri razvoju bolj robustne izvorne kode, kjer so posamezne funkcionalnosti aplikacije smiselno razdeljene po več datotekah, kar olajša vzdrževanje. Angular nato v procesu generiranja JavaScript verzije Angular aplikacije združi vse TypeScript datoteke in jih prevede v jezik JavaScript ter shrani v enotno datoteko.

Trenutno smo aplikacijo pognali s pomočjo ukaza ng serve, ki poskrbi, da se proces generiranja Angular aplikacije v jezik JavaScript izvede v pomnilniku, zato JavaScript datotek ne bomo našli nikjer v mapah projekta. Če bi želeli generirati Angular aplikacijo v jeziku JavaScript, bi uporabili ukaz ng build. Za potrebe razvoja je uporaba ng serve idealna, saj se spremembe spletne strani samodejno naložijo ob spremembi izvorne kode.

import { Pipe, PipeTransform } from "@angular/core";
@Pipe({
  name: "distance",
})
export class DistancePipe implements PipeTransform {
  transform(
    distance: number value: unknown, ...args: unknown[]
  ): string unknown {
    let unit = "m";
    let nDigits = 0;
    if (distance > 1000) {
      distance = distance / 1000;
      unit = "km";
      nDigits = 2;
    }
    return (
      distance.toLocaleString("sl-SI", { maximumFractionDigits: nDigits }) +
      " " +
      unit
    );
    return null;
  }
}

import { Injectable } from "@angular/core";

import { Observable, throwError } from "rxjs";
import { catchError, retry } from "rxjs/operators";
import { Location } from "./location-list/location-list.component";
@Injectable({
  providedIn: "root",
})
export class DemoDataService {
  constructor(private http: HttpClient) {}
  private apiUrl = "http://localhost:3000/api";
  public getLocations(
    lng: number,
    lat: number,
    distance: number,
    nResults: number
  ): Observable<Location[]> {
    const url: string = `${this.apiUrl}/locations/distance?lng=${lng}&lat=${lat}&distance=${distance}&nResults=${nResults}`;
    return this.http
      .get<Location[]>(url)
      .pipe(retry(1), catchError(this.handleError));
  }
  private handleError(error: HttpErrorResponse) {
    return throwError(() => error.error.message || error.statusText);
  }
}

8.2 Izdelava Angular SPA aplikacije na eni strani

Na sliki 8.1 je prikazano, na kateri element naše aplikacije bomo osredotočeni v nadaljevanju. Z uporabo Angular-ja bomo pretvorili obstoječo aplikacijo v aplikacijo na eni strani.

Slika 8.1: Uporaba Angular za pretvorbo obstoječe aplikacije v SPA

Pri pretvorbi obstoječe aplikacije v aplikacijo na eni strani bomo najprej posodobili obstoječo Express aplikacijo, da bo le-ta vsebovala odjemalski del, in bomo šele nato začeli s konstruiranjem aplikacije na eni strani. V tem procesu bomo dodajali vse potrebne Angular gradnike ter predstavili različne dobre prakse pri razvoju.

8.2.1 Prehod iz Express usmerjanja na Angular usmerjanje

Ko se uporabnik pri uporabi aplikacije na eni strani prestavi iz enega zaslona na drugega, ni treba ponovno naložiti celotne strani, kar je ena izmed ključnih prednosti takšnega razvoja. Brskalniku tako ob vsakem uporabniškem kliku ni treba izvesti nove zahteve do strežnika.

Pri trenutni aplikaciji se ob vsakem uporabniškem vnosu URL naslova pošlje zahteva do strežnika, ki jo le-ta obdela in posreduje kot odgovor celotno spletno stran (glej 1. scenarij na sliki 8.2).

Slika 8.2: Različni načini obdelave zahteve pri strežniški aplikaciji in pri različnih scenarijih SPA aplikacije

Pri SPA aplikaciji se pri prvem obisku spletne strani brskalniku posreduje celotna aplikacija, ki se izvaja na strani odjemalca. Prav tako se pri odjemalcu izvaja obdelava zahtev in prikazovanje strani (glej 2. scenarij na sliki 8.2). Pri nadaljnjih zahtevah je zato večina procesiranja izvedenega na strani odjemalca (glej 3. scenarij na sliki 8.2).

8.2.1.1 Odstranitev Express usmerjanja

Kot je prikazano na sliki 8.2, se usmerjanje zahtev s strani odjemalca obdela na strežniški strani v okviru Express aplikacije. Usmerjanje pri Express aplikaciji je bilo implementirano v usmerjevalniku, kjer smo opredelili preslikovalno tabelo, tj. katera metoda krmilnika bo obdelala posamezno spletno zahtevo. V okviru Angular aplikacije bomo naredili podobno, s to razliko, da bomo uporabili usmerjevalnik v Angularju.

Ključna sprememba pri delovanju Angular usmerjevalnika je v tem, da je celotna aplikacija že naložena v spletnem brskalniku, tako da pri uporabniški navigaciji po strani ni treba strani ponovno zahtevati s strežnika. Navigacija po strani postane tako veliko hitrejša in prijetnejša za uporabnika, saj mora uporabnik čakati zgolj na rezultate REST API klicev na strežnik.
